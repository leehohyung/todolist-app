### SC-01 — 회원가입

| 항목 | 내용 |
|---|---|
| **시나리오 ID** | SC-01 |
| **관련 UC** | UC-01 회원가입 |
| **페르소나** | P-01 김민준 |
| **목표** | 이메일과 비밀번호로 TodoListApp 계정을 생성한다. |

**사전 조건**

- 사용자는 TodoListApp에 계정이 없다.
- 브라우저에서 TodoListApp 회원가입 페이지에 접근 가능한 상태이다.

**주요 흐름 (Happy Path)**

1. 김민준은 데스크톱 브라우저에서 TodoListApp 시작 화면에 접근한다.
2. "회원가입" 버튼을 클릭하여 회원가입 폼 화면으로 이동한다.
3. 이름 입력란에 "김민준"을 입력한다.
4. 이메일 입력란에 본인 이메일 주소를 입력한다.
5. 비밀번호 입력란에 비밀번호를 입력하고 비밀번호 확인란에 동일 값을 재입력한다.
6. "가입하기" 버튼을 클릭한다.
7. 시스템은 이메일 중복 여부를 검증하고 비밀번호를 bcrypt로 암호화하여 사용자 계정을 생성한다.
8. 가입 완료 후 시스템은 로그인 화면으로 자동 이동시키며 "회원가입이 완료되었습니다" 메시지를 표시한다.

**예외 흐름**

- **EX-01 이메일 중복**: 이미 사용 중인 이메일을 입력한 경우, 시스템은 "이미 사용 중인 이메일입니다" 오류 메시지를 이메일 입력란 아래에 표시하고 가입을 차단한다.
- **EX-02 비밀번호 불일치**: 비밀번호와 비밀번호 확인이 다른 경우, 시스템은 "비밀번호가 일치하지 않습니다" 오류 메시지를 표시하고 폼 제출을 막는다.

**사후 조건**

- 사용자 계정이 DB에 생성되고 비밀번호는 암호화된 상태로 저장된다.
- 사용자는 해당 이메일과 비밀번호로 로그인할 수 있다.

### SC-02 — 로그인

| 항목 | 내용 |
|---|---|
| **시나리오 ID** | SC-02 |
| **관련 UC** | UC-02 로그인 |
| **페르소나** | P-03 박성호 |
| **목표** | 이메일과 비밀번호로 인증 후 JWT 토큰을 발급받아 앱을 사용할 수 있는 상태가 된다. |

**사전 조건**

- 박성호는 TodoListApp에 이미 계정이 존재한다.
- 모바일 브라우저에서 로그인 화면에 접근한 상태이다.

**주요 흐름 (Happy Path)**

1. 박성호는 모바일 브라우저에서 TodoListApp 로그인 화면을 연다.
2. 이메일 입력란에 가입 시 등록한 이메일을 입력한다.
3. 비밀번호 입력란에 비밀번호를 입력한다.
4. "로그인" 버튼을 탭한다.
5. 시스템은 이메일과 비밀번호를 검증하고 JWT Access Token(유효기간 1시간)과 Refresh Token(유효기간 7일)을 발급한다.
6. 시스템은 할일 목록 화면으로 이동시키고 박성호의 이름을 화면 상단에 표시한다.

**예외 흐름**

- **EX-01 이메일 또는 비밀번호 불일치**: 시스템은 "이메일 또는 비밀번호가 올바르지 않습니다" 메시지를 표시한다. 구체적으로 어떤 항목이 틀렸는지는 보안상 알려주지 않는다.
- **EX-02 Access Token 만료**: 앱 사용 중 토큰이 만료된 경우, 시스템은 Zustand 메모리에 보관된 Refresh Token으로 Access Token을 자동 재발급하여 사용자가 재로그인 없이 계속 사용할 수 있도록 한다.

**사후 조건**

- JWT Access Token과 Refresh Token이 Zustand 메모리(in-memory)에 저장되어 이후 API 요청 시 Authorization 헤더에 포함된다. localStorage나 Cookie에는 저장되지 않는다.
- 사용자는 인증된 상태로 할일 목록 화면에 진입한다.
- 페이지를 새로고침하면 메모리가 초기화되어 로그인 화면으로 이동한다.

### SC-03 — 개인정보 수정

| 항목 | 내용 |
|---|---|
| **시나리오 ID** | SC-03 |
| **관련 UC** | UC-03 개인정보 수정 |
| **페르소나** | P-01 김민준 |
| **목표** | 마이페이지에서 자신의 이름 또는 비밀번호를 변경한다. |

**사전 조건**

- 김민준은 로그인된 상태이다.
- 마이페이지 화면에 접근한 상태이다.

**주요 흐름 (Happy Path)**

1. 김민준은 화면 상단 또는 메뉴에서 마이페이지로 이동한다.
2. 현재 이름("김민준")이 입력란에 표시된 것을 확인한다.
3. 이름 입력란의 값을 "김민준 팀장"으로 수정한다.
4. "저장" 버튼을 클릭한다.
5. 시스템은 JWT 토큰으로 본인 여부를 검증한 후 이름을 업데이트한다.
6. 화면 상단의 사용자 이름이 "김민준 팀장"으로 즉시 반영된다.
7. "정보가 수정되었습니다" 성공 메시지가 표시된다.

**예외 흐름**

- **EX-01 이름 공백 입력**: 이름을 비워두고 저장하려 하면 시스템은 "이름은 필수 입력 항목입니다" 오류 메시지를 표시하고 저장을 막는다.
- **EX-02 비밀번호 변경 시 현재 비밀번호 불일치**: 비밀번호를 변경하려는 경우 현재 비밀번호 확인 단계에서 불일치하면 "현재 비밀번호가 올바르지 않습니다" 메시지를 표시한다.

**사후 조건**

- DB의 사용자 이름 또는 비밀번호가 업데이트된다.
- 비밀번호 변경 시 새 비밀번호는 bcrypt로 암호화되어 저장된다.

### SC-04 — 카테고리 추가

| 항목 | 내용 |
|---|---|
| **시나리오 ID** | SC-04 |
| **관련 UC** | UC-04 카테고리 추가 |
| **페르소나** | P-02 이수연 |
| **목표** | 기본 카테고리(업무, 개인, 쇼핑) 외에 자신만의 "사이드프로젝트" 카테고리를 생성한다. |

**사전 조건**

- 이수연은 로그인된 상태이다.
- 카테고리 관리 화면에 접근한 상태이다.
- 현재 기본 카테고리(업무, 개인, 쇼핑)만 존재한다.

**주요 흐름 (Happy Path)**

1. 이수연은 카테고리 관리 화면에서 기본 카테고리 목록(업무, 개인, 쇼핑)을 확인한다.
2. "카테고리 추가" 버튼을 탭한다.
3. 카테고리 이름 입력란에 "사이드프로젝트"를 입력한다.
4. "추가" 버튼을 탭한다.
5. 시스템은 JWT 토큰으로 인증을 확인하고 해당 사용자 소유의 카테고리를 DB에 생성한다.
6. 카테고리 목록에 "사이드프로젝트"가 즉시 추가되어 표시된다.

**예외 흐름**

- **EX-01 카테고리명 공백**: 이름을 입력하지 않고 추가하려 하면 "카테고리 이름을 입력해 주세요" 오류 메시지가 표시된다.
- **EX-02 비인증 접근**: 토큰이 없는 상태에서 API를 직접 호출하면 시스템은 HTTP 401을 반환한다.

**사후 조건**

- 이수연 소유의 "사이드프로젝트" 카테고리가 DB에 저장된다.
- 카테고리 목록 조회 시 기본 카테고리와 함께 새 카테고리가 조회된다.
- 할일 등록 시 "사이드프로젝트" 카테고리를 선택할 수 있다.

### SC-05 — 카테고리 삭제

| 항목 | 내용 |
|---|---|
| **시나리오 ID** | SC-05 |
| **관련 UC** | UC-05 카테고리 삭제 |
| **페르소나** | P-02 이수연 |
| **목표** | 더 이상 사용하지 않는 카테고리를 삭제하고, 해당 카테고리의 할일이 기본 카테고리로 자동 이동되는 것을 확인한다. |

**사전 조건**

- 이수연은 로그인된 상태이다.
- "학습" 카테고리(이수연 소유)가 존재하며 해당 카테고리에 할일이 3건 등록되어 있다.

**주요 흐름 (Happy Path)**

1. 이수연은 카테고리 관리 화면에서 "학습" 카테고리 옆의 삭제 버튼을 탭한다.
2. 시스템은 "해당 카테고리의 할일 3건이 '개인' 카테고리로 이동됩니다. 삭제하시겠습니까?" 확인 다이얼로그를 표시한다.
3. 이수연은 "삭제" 버튼을 탭하여 확인한다.
4. 시스템은 "학습" 카테고리를 DB에서 삭제하고, 해당 카테고리에 속한 할일 3건의 카테고리를 기본 카테고리 "개인"으로 자동 변경한다.
5. 카테고리 목록에서 "학습"이 제거된다.
6. "카테고리가 삭제되었습니다" 메시지가 표시된다.

**예외 흐름**

- **EX-01 기본 카테고리 삭제 시도**: 기본 카테고리(업무, 개인, 쇼핑)의 삭제 버튼은 비활성화 상태로 표시되어 클릭 자체가 불가하다.
- **EX-02 타인 카테고리 삭제 시도**: 다른 사용자 소유의 카테고리 ID로 API를 호출하면 시스템은 HTTP 403을 반환한다.

**사후 조건**

- "학습" 카테고리가 DB에서 삭제된다.
- 기존 "학습" 카테고리에 속했던 할일 3건의 category_id가 기본 카테고리 "개인"의 ID로 변경된다.

### SC-06 — 할일 등록

| 항목 | 내용 |
|---|---|
| **시나리오 ID** | SC-06 |
| **관련 UC** | UC-06 할일 등록 |
| **페르소나** | P-03 박성호 |
| **목표** | 고객사 방문 후 후속 할일을 즉시 모바일로 등록한다. |

**사전 조건**

- 박성호는 로그인된 상태이다.
- 할일 목록 화면에 있다.

**주요 흐름 (Happy Path)**

1. 박성호는 고객사 미팅을 마치고 이동 중 모바일 브라우저로 앱을 연다.
2. 할일 목록 화면 하단의 "+" 버튼 또는 "할일 추가" 버튼을 탭한다.
3. 할일 등록 폼(모달 또는 슬라이드 패널)이 열린다.
4. 제목 입력란에 "A사 견적서 발송"을 입력한다.
5. 카테고리 선택란에서 "업무"를 선택한다.
6. 설명 입력란에 "이메일로 PDF 형식으로 발송, 담당자 김대리 참조"를 입력한다.
7. 종료 예정일 선택기에서 "2026-05-15"를 선택한다.
8. "등록" 버튼을 탭한다.
9. 시스템은 할일을 DB에 저장하고 목록 화면으로 돌아간다.

### SC-10 — 할일 목록 조회 및 필터

9. 필터를 초기화하면 전체 할일 목록이 다시 표시된다.

**예외 흐름**

- **EX-02 잘못된 날짜 범위**: 시작일이 종료일보다 늦은 경우 "시작일은 종료일보다 빠른 날짜여야 합니다" 오류 메시지를 표시하고 조회를 막는다.

**사후 조건**

- 화면에는 적용된 필터 조건이 표시된 상태로 해당 조건에 맞는 할일 목록이 표시된다.
- 필터 상태는 페이지 이탈 전까지 유지된다.

## 3. 통합 시나리오

### TS-01. 김민준의 첫 일주일: TodoListApp 도입부터 루틴 정착까지

---

**월요일 오전 — 가입과 첫 로그인**

월요일 아침 출근 직후, 마케팅 팀장 김민준은 팀 단톡방에서 TodoListApp 링크를 받았다. 지난주부터 업무 할일과 개인 일정이 뒤섞여 중요한 보고 마감을 하마터면 놓칠 뻔했던 터라 한번 써보기로 마음먹었다.

데스크톱 브라우저에서 링크를 열자 깔끔한 회원가입 화면이 나타났다. 이름("김민준"), 이메일, 비밀번호를 입력하고 가입 버튼을 눌렀다. 잠시 후 "회원가입이 완료되었습니다" 메시지와 함께 로그인 화면으로 이동했다. 이메일과 비밀번호를 입력하자 곧바로 할일 목록 화면이 열렸다. 화면 상단에 "안녕하세요, 김민준님"이라는 환영 문구가 반겨주었다.

---

**월요일 오전 — 카테고리 정리**

할일 목록 화면은 비어 있었지만 우선 카테고리 구조부터 정리하기로 했다. 카테고리 관리 화면에 들어가자 기본 카테고리 세 개(업무, 개인, 쇼핑)가 이미 제공되어 있었다. "쇼핑은 나한테 필요 없겠지만, 업무와 개인은 딱이네." 김민준은 추가로 "사내 교육"이라는 카테고리를 만들기로 했다. 카테고리 추가 버튼을 클릭하고 "사내 교육"을 입력하자 목록에 즉시 추가되었다.

---

**월요일 오전 — 이번 주 할일 일괄 등록**

이제 본격적으로 할일을 등록했다. 할일 추가 버튼을 눌러 등록 폼을 열었다.

첫 번째로 "Q2 캠페인 기획서 작성"을 입력했다. 카테고리는 "업무", 종료 예정일은 이번 금요일(2026-05-17)로 설정하고 설명란에 "임원 보고 전 초안 완료 필요"를 적었다. 등록 버튼을 누르자 목록에 바로 나타났다.

이어서 "5월 팀 회식 장소 예약"(카테고리: 업무, 종료 예정일: 2026-05-14), "신입 온보딩 자료 배포"(카테고리: 사내 교육, 종료 예정일: 2026-05-13), "생일 선물 구매"(카테고리: 개인, 종료 예정일: 2026-05-16)를 차례로 등록했다. 4개의 할일이 목록에 채워지자 오늘 해야 할 일이 한눈에 들어왔다.

---

**월요일 오후 — 완료 처리**

점심 식사 후 돌아온 김민준은 오전에 신입 온보딩 자료를 이미 배포했다는 것을 기억했다. 목록에서 "신입 온보딩 자료 배포" 항목의 체크박스를 클릭했다. 항목에 취소선이 그어지며 완료 배지가 붙었다. 시스템이 완료 일시를 자동으로 기록하고 있다는 사실은 모르지만, 깔끔하게 처리된 느낌이 좋았다.

---

**화요일 — 기한 초과 항목 발견**

화요일 아침 앱을 열었더니 "5월 팀 회식 장소 예약" 항목에 붉은 색상의 "기한 초과" 배지가 붙어 있었다. 종료 예정일이 2026-05-14인데 어제 처리하지 못했던 것이다. 김민준은 바로 전화로 예약을 완료한 뒤 체크박스를 클릭하여 완료 처리했다. 배지가 사라지고 완료 상태로 전환되었다.

---

**수요일 — 모바일에서 빠른 할일 추가**

수요일에는 외부 미팅이 있었다. 이동 중 모바일 브라우저로 앱을 열어 로그인했다. 미팅에서 새로운 요청을 받아 "B사 제안서 수정본 전달"(카테고리: 업무, 종료 예정일: 2026-05-19)을 모바일에서 바로 등록했다. 화면이 모바일에 맞게 최적화되어 있어 입력이 불편하지 않았다.

---

**목요일 — 필터 활용으로 우선순위 파악**

목요일 아침, 업무 카테고리에서 미완료 상태의 항목이 몇 개인지 파악하고 싶었다. 필터 바에서 카테고리를 "업무"로, 완료 여부를 "미완료"로 설정한 뒤 적용 버튼을 눌렀다. "Q2 캠페인 기획서 작성"과 "B사 제안서 수정본 전달" 두 건이 표시되었다. 금요일 마감인 기획서를 오늘 안에 마무리해야 한다는 것을 직관적으로 확인했다.

---

**금요일 — 기획서 완료 및 한 주 마무리**

금요일 오후, 기획서 초안을 임원에게 공유하고 나서 "Q2 캠페인 기획서 작성" 항목의 체크박스를 클릭했다. 완료 처리가 되며 한 주의 주요 업무가 마무리되었다.

필터를 전체 조회로 되돌리고 목록을 훑어보았다. 완료 항목에는 취소선이, 미완료 항목은 선명하게 강조되어 있었다. 다음 주 마감인 "B사 제안서 수정본 전달"이 남아 있다는 것도 명확하게 확인했다.

---

**일주일 후 — 자연스러운 루틴 정착**

일주일이 지나자 김민준에게 TodoListApp은 출근 직후 열어 오늘의 미완료 항목을 확인하고, 업무 완료 직후 체크하는 자연스러운 루틴이 되었다. 별도 학습 없이 가입부터 필터 활용까지 자연스럽게 진행된 덕분에 도입 장벽이 거의 없었다. "기한 초과" 표시 덕분에 마감을 놓치는 경우도 눈에 띄게 줄었다.
